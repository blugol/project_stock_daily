# [NotebookLM 검토용] 주식 퀀트 매매 자동화 시스템 총결산 및 토의 안건 (상세본)

본 문서는 현재 개발 중인 "HTS 연동형 퀀트(Quant) 매매 보조 시스템"의 기획 의도부터 현재까지의 상세한 개발 과정, 기술 스택, 그리고 현재 직면한 기술적 이슈를 총망라한 문서입니다. 이 방대한 컨텍스트를 바탕으로 AI와 토의하여 완벽한 해결책과 데이터 설계 방향을 도출하고자 합니다.

---

## 1. 프로젝트 기획 및 핵심 비전 (Project Vision)
* **목적**: 트레이더가 키움증권 HTS(영웅문)에서 특정 종목을 클릭할 때마다, 0.1초 만에 백그라운드에서 해당 종목의 '세력 매집 여부, 펀더멘털 안전성, 차트 기술적 타점'을 수학적으로 연산하여 텍스트로 보고해 주는 "초고속 퀀트 보조 뇌(Brain)"를 구축.
* **주요 매매 기법 (주식단테 농사매매 로직 등 반영)**:
  * 차트: 112일선(반년), 224일선(1년), 448일선(2년) 기반의 '밥그릇 패턴' 돌파 포착 및 구름대/볼린저밴드 지지 확인.
  * 재무: 부채비율 150% 이하, 3년 연속 적자 기업 필터링 (깡통/상폐 리스크 원천 차단).
  * 수급: 대주주 지분율 스캔을 통한 유통비율 60~80%의 '세력 빈집' 작전주 판별.
  * 계좌: 현금 비중 30% 감시, -20% 하락 시 1-1-2-4 마틴게일(Martingale) 분할매수 시뮬레이션 연산.
* **불문율(Ground Rule)**: 예쁜 UI/디자인은 절대 배제한다. 오직 '데이터가 정확히 연산되어 들어오는지' 그 기능성에만 집중하여 Raw JSON 형태로 텍스트 출력만 허용한다.

## 2. 지금까지의 상세 개발 프로세스 및 성공 내역 (What We've Done)

### Phase 1: 기반 인프라 구축 및 API 분석
* 트레이더가 발급받은 4개의 API 키(DART API Key, 키움증권 App Key, Secret Key, Account No)를 시스템 환경변수(`.env`)에 안전하게 셋팅 완료.
* 각 키의 역할을 정의함: DART는 기업 재무/공시, 키움은 시세/계좌 연동에 사용.

### Phase 2: 완벽한 "한글 종목명 ↔ 시스템 코드" 번역 사전(Mapping) 구축
HTS 화면의 한글 종목명(예: '삼성전자')을 각 API 서버가 알아들을 수 있는 식별 코드로 변환하기 위해 크롤링 스크립트(`fetch_mappings.py`)를 개발 및 실행.
* **KRX 상장사 매핑**: 한국거래소 전체 2,873개 종목의 한글명과 6자리 종목코드(`005930`)를 매핑하여 `app/stock_codes.json` 생성 완료.
* **DART 고유번호 매핑**: 금감원 오픈 API 마스터 XML을 다운받아 상장사 3,945개의 한글명과 8자리 고유번호(`00126380`)를 매핑하여 `app/dart_codes.json` 생성 완료.

### Phase 3: HTS 텍스트 감지 스나이퍼 모듈 개발
* 파이썬의 `win32gui` 라이브러리를 활용하여, 윈도우 OS에서 실행 중인 **키움증권 HTS(영웅문)의 윈도우 창 제목(Title)을 실시간으로 후킹(Hooking)** 하는 `server.py` 로직 개발 완료.
* 메뉴 이름이나 계좌번호 등 불필요한 텍스트를 걸러내는 강력한 정규식(Regex) 및 50여 개의 금지어 사전(`IGNORE_EXACT`) 필터링 알고리즘 탑재. 오직 순수한 '종목명'만 감지해 냄.

### Phase 4: API 클라이언트 및 퀀트 연산 엔진(Quant Engine) 개발
종목 감지 시 비동기(Async)로 데이터를 수집하고 수학적으로 연산하는 `api_client.py` 개발 완료.
* **DART 클라이언트 (`DartClient`)**: 성공적으로 작동 중. DART 고유번호를 던져서 최근 90일 이내의 분기보고서, 유상증자 등 주요 공시 5건을 정상적으로 긁어옴.
* **퀀트 엔진 (`QuantClient`)**: 성공적으로 작동 중. `FinanceDataReader` 라이브러리로 해당 종목의 과거 900일 치 일봉(OHLCV)을 네트워크로 긁어온 뒤, `pandas-ta` 라이브러리를 통해 **112일/224일/448일 이평선, 볼린저밴드 하단선, 일목균형표(구름대) 선행스팬 A/B 값**을 파이썬 내부에서 소수점까지 오차 없이 완벽하게 계산해 냄.
* **키움증권 클라이언트 (`KiwoomClient`)**: OAuth2 규격에 맞춰 키움증권 실전투자 서버(`api.kiwoom.com`)에 접근하여 24시간 유효 Access Token을 발급받는 데까지 완벽하게 성공함. (※ 시세 조회 에러는 하단 이슈 참조)

### Phase 5: 비동기 웹소켓 서버 및 프론트엔드 연동
* 파이썬 `asyncio`와 `websockets`를 활용하여, HTS 종목 변경이 감지되는 즉시 위 3개의 클라이언트(DART, 퀀트, 키움)를 병렬(`asyncio.gather`)로 동시 실행시킴.
* 수집 및 연산된 모든 결과값(Payload)을 웹 브라우저(`index.html`)로 전송. 브라우저는 어떠한 디자인 요소도 없이 순수 텍스트(Raw JSON) 포맷으로만 화면에 데이터를 즉시 뿌려주는 상태 (원칙 100% 준수).

## 3. 핵심 토의 안건 및 해결 요망 사항 (Discussion Points)
NotebookLM은 지금까지의 이 거대한 시스템 아키텍처와 흐름을 딥러닝하고, 다음 두 가지 기술/기획 이슈에 대한 명확한 해답을 제시해 주시기 바랍니다.

### Issue A: 키움증권 REST API '현재가 조회' 500 에러 원인 및 TR 양식 분석
* **상황**: 발급받은 키움증권 토큰으로 '국내주식 현재가 조회' 엔드포인트를 찔렀으나 `500 INTERNAL_SERVER_ERROR (예기치 못한 에러가 발생하였습니다)`가 반환됨. (DART와 퀀트 엔진은 문제없이 정상 작동)
* **원인 추정**: 현재 코드에서 한국투자증권의 TR ID인 `FHKST01010100`을 그대로 차용하여 파라미터(`fid_cond_mrkt_div_code`, `fid_input_iscd`)를 넘겼으나, **키움증권 서버에서 요구하는 특정 TR ID 양식이나 필수 헤더(예: Mac Address 등)가 누락**되었을 확률이 높음.
* **NotebookLM에게 요청하는 답변**: 키움증권 OpenAPI/REST API 공식 문서나 최신 개발자 가이드를 바탕으로, **국내주식 단일 종목의 현재가(시세)를 REST API로 조회할 때 정확히 어떤 TR ID 값과 파라미터 구조(JSON/Headers)를 사용해야 하는지** 디버깅하여 찾아주세요.

### Issue B: 농사매매 최적화 데이터 필드(Data Schema) 추출 설계
* **상황**: 퀀트 연산과 API를 통해 엄청난 양의 데이터(차트, 재무, 공시)를 긁어올 수 있는 뼈대가 완성되었습니다. 이제 이 중에서 트레이더가 화면에서 봐야 할 '알짜배기 데이터'만 골라낼 차례입니다.
* **NotebookLM에게 요청하는 답변**: 성공적인 단테 기법(농사매매, 빈집털이) 및 마틴게일 자금 관리를 위해, 반드시 화면(텍스트)에 띄워야 하는 **데이터 항목(필드) 리스트를 정의해 주세요.** 
* *(예시: 현재가, 전일대비 등락률, 224일선 대비 이격률, DART 부채비율, 자본총계 대비 대주주 지분율, 통계좌 현금 비중, 마틴게일 2차 진입 요구 예수금 등)*
